# ORCA 项目三人分层分工表、本周任务与接口约定

## 1. 使用说明

这份文档可以直接发给队友。

用途只有三个：

1. 让三个人先选层
2. 让每个人明确本周要交什么
3. 让后面联调时少扯皮

---

## 2. 三人分层分工表

### 2.1 1号位：手部硬件与 `orca_core` 负责人

定位：

1. 负责设备端
2. 负责手部执行层
3. 负责把手安全、稳定地跑起来

负责范围：

1. Ubuntu / Linux 基本环境
2. 串口、驱动、权限
3. `orca_core` 项目复现
4. `config.yaml` 理解与核对
5. 标定、回中、动作测试
6. 基础手势脚本

必须交付：

1. 手能连接
2. 能读取状态
3. 能安全标定
4. 能执行至少 `2` 个基础手势
5. 能写出一个最小控制脚本

这个位置适合：

1. 最能排查设备问题的人
2. 最愿意啃底层的人
3. 最不怕联调的人

### 2.2 2号位：ROS 2 与系统集成负责人

定位：

1. 负责中间层
2. 负责把上层命令变成对 `orca_core` 的调用
3. 负责系统能串起来

负责范围：

1. ROS 2 基础环境
2. ROS 2 package
3. hand node
4. topic / service / action 设计
5. 状态发布
6. 联调流程与状态机

必须交付：

1. 一个最小手部 ROS 2 节点
2. 一个最小 service 或 topic 接口
3. 一条“命令下发 -> 手部执行 -> 状态返回”的链路
4. 一张系统通信图

这个位置适合：

1. 最适合做中间协调的人
2. 最适合做系统中枢的人
3. 最适合做队长的人

### 2.3 3号位：算法与展示应用负责人

定位：

1. 负责上层应用
2. 负责让系统有“抓什么、怎么抓、怎么展示”的能力
3. 负责比赛表现力

负责范围：

1. 相机接入或输入模拟
2. 目标识别或规则判断
3. 抓取策略选择
4. 手势选择逻辑
5. 页面 / 看板 / 演示逻辑
6. 答辩展示素材

必须交付：

1. 一个最小输入源
2. 一个最小决策逻辑
3. 一个最小展示界面或展示页面
4. 一个清晰的演示脚本

这个位置适合：

1. 最会讲清楚方案的人
2. 最适合做应用层封装的人
3. 最适合兼顾比赛展示的人

## 3. 三层关系图

建议你们统一按下面这个结构理解：

1. 硬件层  
   灵巧手本体、舵机、串口、电源、相机
2. 执行控制层  
   `orca_core`
3. 系统集成层  
   `ROS 2`
4. 算法与应用层  
   识别、策略、展示、比赛流程

三个人对应关系建议是：

1. 1号位负责 `硬件层 + 执行控制层`
2. 2号位负责 `系统集成层`
3. 3号位负责 `算法与应用层`

## 4. 每个人本周任务

### 4.1 1号位本周任务

第一个目标：

1. 把 `orca_core` 看明白
2. 让手能执行最小动作

本周任务：

1. 阅读 `config.yaml`

4. 阅读 `scripts/simple_test.py`
5. 阅读 `SAFE_CALIBRATION_GUIDE.md`

7. 写一个两关节最小手势脚本

本周验收：

1. 能解释 `index_mcp`、`index_pip`、`thumb_mcp` 对应哪个电机
2. 能解释 `connect -> enable_torque -> set_joint_positions -> disconnect`
3. 能给出一个最小控制脚本

### 4.2 2号位本周任务

第一个目标：

1. 把系统结构想清楚

本周任务：

5. 读懂 `orca_core` 对外最小调用方式
6. 设计一个 hand node 草图
7. 设计最小命令格式和状态格式

本周验收：

1. 画出一张系统结构图
2. 画出一张最小通信图
3. 写出一版 hand node 接口草案

### 4.3 3号位本周任务

第一个目标：

1. 先有最小应用闭环
2. 不先追求复杂模型

本周任务：

1. 定义输入目标类型
2. 定义输出手势名
3. 做一个规则版决策逻辑
4. 设计展示页原型
5. 设计比赛演示流程
6. 准备状态展示字段

本周验收：

1. 能给出 `输入 -> 手势输出` 规则
2. 能给出一页最小展示界面草图
3. 能说清楚评委演示时看什么

## 5. 三层之间的接口约定

这一部分必须尽早定下来。

否则后面一定会出现：

1. 算法端输出了，设备端接不了
2. ROS 2 定了消息，`orca_core` 不好调
3. 平台端想展示，结果没有统一状态字段

### 5.1 算法层 -> ROS 2 层

算法层不要直接输出电机角度。

建议输出统一的“抓取意图”：

1. `target_type`
2. `target_id`
3. `target_position`
4. `grasp_mode`
5. `confidence`

建议字段示例：

```json
{
  "target_type": "small_block",
  "target_id": "obj_001",
  "target_position": [0.12, 0.05, 0.18],
  "grasp_mode": "pinch",
  "confidence": 0.86
}
```

### 5.2 ROS 2 层 -> `orca_core` 层

ROS 2 层不要直接控制底层电机协议。

建议只把命令转成两种形式之一：

1. `gesture_name`
2. `joint_targets`

建议字段示例：

```json
{
  "gesture_name": "pinch"
}
```

或者：

```json
{
  "joint_targets": {
    "thumb_mcp": 20,
    "index_mcp": 35,
    "index_pip": 50
  }
}
```

### 5.3 `orca_core` 层 -> ROS 2 层

`orca_core` 返回给 ROS 2 的状态建议统一成：

1. `connected`
2. `calibrated`
3. `current_action`
4. `joint_state`
5. `motor_current`
6. `motor_temperature`
7. `error_code`
8. `error_message`

建议字段示例：

```json
{
  "connected": true,
  "calibrated": true,
  "current_action": "pinch",
  "error_code": 0,
  "error_message": ""
}
```

### 5.4 ROS 2 层 -> 平台展示层

平台端不要直接读底层串口或电机寄存器。

建议平台端只消费统一状态：

1. 当前目标
2. 当前动作
3. 执行状态
4. 在线状态
5. 最近一次结果
6. 告警信息

---

## 6. 最小消息命名建议

为了减少后期改名成本，建议先统一英文 snake_case。

建议命名：

1. `target_type`
2. `target_position`
3. `grasp_mode`
4. `gesture_name`
5. `joint_targets`
6. `current_action`
7. `execution_status`
8. `error_message`

不建议混用：

1. 一会儿中文
2. 一会儿拼音
3. 一会儿 camelCase
4. 一会儿手写缩写

---

## 7. 本周联调目标

本周不要追求完整工业系统。

本周只追求一个最小闭环：

1. 输入一个目标或按钮
2. 系统给出一个手势名
3. ROS 2 或中间层把命令转发给手部
4. `orca_core` 执行动作
5. 页面显示动作结果

只要这条链能跑通，你们就有比赛工程基础了。

---

## 8. 建议的选人标准

如果现在要让大家自己选，建议按下面标准：

1. 最会排错的人选 `1号位`
2. 最会串系统的人选 `2号位`
3. 最会做应用和展示的人选 `3号位`

如果你是队长，建议优先选 `2号位`。

原因：

1. 这个位置最像中枢
2. 能同时理解上下游
3. 方便你推进整体节奏
